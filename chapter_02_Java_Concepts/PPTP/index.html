05 - Comments in JavaScript

05_Core_Comments_JS.js Comments

JavaScript supports single-line comments (//) and multi-line comments (/* */ or /** */). Comments are ignored by the engine and are used to document code, note authorship, or temporarily disable code.

// This is comment. -This code will not be execute
console.log("Hello");
/**
 *  This is multi line
 *  Author : Prrmmod Dutta
 *  Date : 14-Feb-2026
 **/

/*
 *  This is multi line
 *  Author : Prrmmod Dutta
 *  Date : 14-Feb-2026
 */

Covers the three ways to declare variables: var (function-scoped, allows re-declaration), let (block-scoped, no re-declaration in same scope), and const (block-scoped, cannot be reassigned). Shows scoping differences with practical examples inside functions and if blocks.

09 - Hoisting Introduction

09_Hoisting.js Hoisting

Hoisting is JavaScript's behavior of moving declarations to the top during the compilation phase. With var, the declaration is hoisted with a value of undefined. The code is NOT physically moved -- it is a mental model for understanding the JS engine's two-phase process: Memory Creation and Execution.

// ---- Phase 1: Memory Creation ----
// var a  = undefined;
// var b  = undefined;

// ---- Phase 2: Execution ----
console.log(a); // undefined
var a = "Pramod";
console.log(a); // changed

// Hoisting does NOT physically move your code.
// It is a mental model to understand how the JS engine
// handles declarations during compilation.

10 - Hoisting with var

10_hoisting_var.js Hoisting

Demonstrates how var declarations are hoisted to the top of their scope with the initial value of undefined. The assignment stays in place during execution.

console.log(greeting);   // Output: undefined
var greeting = "Hello!";
console.log(greeting);   // Output: "Hello!"

// Behind the scenes:
// var greeting;              <-- hoisted with undefined
// console.log(greeting);    <-- undefined
// greeting = "Hello!";      <-- assignment stays in place
// console.log(greeting);    <-- "Hello!"

14 - let Hoisting and TDZ

14_let_hoisting.js TDZ

Unlike var, let declarations are hoisted but NOT initialized. Accessing them before declaration causes a ReferenceError due to the Temporal Dead Zone (TDZ). The code runs until it hits the let access in the TDZ.

// console.log(username);
console.log("Pramod is awesome");
console.log("Pramod is awesome");
console.log("Pramod is awesome");
console.log("Pramod is awesome");
let username = "Dutta";
console.log("Pramod is awesome");
console.log("Pramod is awesome");
console.log("Pramod is awesome");
console.log(username);

15 - let Block Scope

15_let_block.js Block Scope

Demonstrates that let is block-scoped. Inside the if block, accessing a let variable before its declaration (even if a variable with the same name exists in the outer scope) triggers the TDZ error.

let a = "Pramod";

if(true){
    console.log(a); //local variable, TDZ
    let a = "temp";
}

16 - var in if/loop

16_var_if_loop.js Scope

Shows that var is NOT block-scoped. Inside an if block, var re-declares the same variable in the function/global scope, so it modifies the outer variable rather than creating a new one.

var a = "Pramod";

if(true){
    console.log(a);
    var a = "temp";
    console.log(a);
}

17 - TDZ Deep Dive

17_rogit.js TDZ

Another example illustrating the Temporal Dead Zone with let. Even though a is declared in the outer scope, the let a = 20 inside the if block creates a new block-scoped variable, and accessing it before the declaration throws an error.

let a = 10
console.log(a)
if (true){
    console.log(a); //TDZ
    let a = 20;
}

18 - const Keyword

18_const.js Constants

The const keyword declares block-scoped variables that cannot be reassigned after initialization. It also experiences TDZ like let. Ideal for values that should never change, like API URLs, mathematical constants, or configuration values.
